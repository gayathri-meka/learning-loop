Building blocks 01–04

Routing & Access

The front door of every web system. These components handle how requests find your servers, how traffic is distributed, how content is delivered fast, and how abuse is prevented.

⌕
01
DNS — Domain Name System
Translates human-readable names into IP addresses
routing pm relevant sa relevant
⌄
What it is
A globally distributed phone book. When a browser visits sensai.in, DNS translates that name into an IP address (e.g. 76.76.21.21). Governed by ICANN, operated by a hierarchy: root servers → TLD registries (Verisign owns .com) → registrars (GoDaddy, Namecheap) → your authoritative nameserver (Cloudflare, Vercel).
What problem it solves
Humans can't memorize IP addresses. Servers change IPs when you migrate hosts. DNS lets you decouple the public name from the underlying infrastructure — change where your server lives without changing what users type.
Key concept: TTL
Time-To-Live controls how long resolvers cache a record. High TTL (86400s = 1 day) = changes are slow to propagate. Low TTL (300s = 5 min) = changes propagate fast but more lookup traffic. Lower TTL before migrations; raise it after.
Record types
A record: name → IP address
CNAME: name → another name (e.g. → Vercel)
MX: mail routing
TXT: domain verification, SPF/DKIM for email
NS: which nameservers are authoritative
When it comes up in conversation
  • Customer wants app.theirdomain.com instead of your default URL → "That's a CNAME record, takes 10 minutes to configure, propagates within the hour."
  • Post-migration: "Why are some users still hitting the old server?" → "Their resolver cached the old TTL. Should clear within [TTL] seconds."
  • Adding a new service: api.sensai.in → a CNAME pointing to your backend. No new domain purchase needed.
  • Incident war room: engineer says "it's a DNS misconfiguration" → you know this means propagation delay, not a code fix, and can set customer expectations accordingly.
  • Enterprise customer asks "can we keep our domain?" → yes, custom domain = DNS record change on their side, 15-minute engineering task.
SA depth: Understand GeoDNS (routing users to nearest region by IP location) and how moving nameservers to Cloudflare gives you DDoS protection + analytics for free, separate from just adding a CNAME.
02
Load Balancer
Distributes incoming traffic across multiple servers
routing sa relevant
⌄
What it is
A server that sits in front of your application servers and decides which one handles each incoming request. Think: maître d' at a restaurant with many tables, routing each party to an available one. Without it, all traffic hits one server until it falls over.
What problem it solves
Single point of failure: if one server dies, traffic routes to healthy ones.
Capacity: spread 100K req/sec across 20 servers instead of crushing one.
Zero-downtime deploys: update servers one at a time while the LB keeps traffic flowing.
L4 vs L7 (important for SAs)
L4 (transport): routes by IP/port only. Fast, dumb. Doesn't see request content.
L7 (application): routes by URL path, headers, cookies. sensai.in/api/* → API servers. /static/* → CDN. /ws/* → WebSocket servers. This is what you'd configure in practice.
Algorithms
Round robin: 1→2→3→1→2→3. Simple.
Least connections: next request → least busy server. Smarter.
Consistent hashing: same user always hits same server (critical for stateful services, distributed caches).
When it comes up in conversation
  • Customer asks "how do you ensure uptime if a server goes down?" → Load balancer with health checks removes unhealthy servers automatically.
  • Capacity planning: "can you handle 10× traffic?" → Add servers behind the LB; it distributes load automatically. Horizontal scaling.
  • Feature request: "can different customers be on isolated infrastructure?" → LB can route by subdomain or header to dedicated server pools.
  • Engineer mentions "we need sticky sessions" → They mean consistent hashing / session affinity. Relevant if your app stores state in memory (not ideal, but common).
  • Discussing SLAs: LB + health checks is the mechanism behind "we automatically failover in under 30 seconds."
SA depth: In a three-tier architecture (web → app → DB), you need a load balancer at each tier boundary. The three-LB diagram isn't over-engineering — it's what lets you scale each tier independently. AWS ALB (L7), NLB (L4), and Vercel's edge are all load balancers you'll reference in customer conversations.
03
CDN — Content Delivery Network
Serves content from servers physically close to the user
routing pm relevant sa relevant
⌄
What it is
A global network of servers (Points of Presence / PoPs) that cache copies of your content close to users. Cloudflare has 300+ PoPs globally, many inside ISP facilities. Latency is physical — Mumbai to San Francisco is min 140ms round-trip regardless of server speed. CDN solves this by serving from a nearby PoP (8ms instead of 140ms).
What problem it solves
Latency: physics is the limit — CDN moves data closer.
Origin load: 95% of requests served from edge, origin server barely touched.
Cost: CDN bandwidth is cheaper than origin server bandwidth at scale.
Availability: if origin goes down, CDN can serve stale cache.
What gets cached vs not
Cache (forever): JS bundles, CSS, fonts, images with versioned filenames.
Cache (short TTL): HTML pages, OG images, semi-static content.
Never cache: API responses, auth tokens, user-specific data, webhooks. A cached API response served to the wrong user = data leak.
Cache invalidation
Versioned filenames (best): bundle.a3f9c2.js — content never changes at that URL, cache forever, new deploy = new URL.
Purge API: Cloudflare/Vercel let you explicitly delete a URL from all PoPs on deploy.
Short TTL: accept staleness up to N minutes.
When it comes up in conversation
  • Customer in India/Africa complains about slow load → "Our assets are on Cloudflare's CDN with PoPs in Mumbai and Nairobi — first visit may be slow (cache miss), repeat visits are <20ms."
  • Planning video/media features → Engineer will say "we need object storage + CDN for this." Cost conversation: CDN bandwidth pricing (Cloudflare R2 is free egress; AWS CloudFront charges per GB).
  • Enterprise customer with strict data residency → "Can our data stay in the EU?" CDN geo-restriction or private CDN configuration is the answer.
  • Post-deploy: "users still see the old version" → cache invalidation issue. Was the deploy purge called? Are filenames versioned?
  • Discussing SensAI's global expansion → CDN means you don't need servers in every region to have fast load times globally.
SA depth: Pull CDN (Vercel/Cloudflare default — fetches from origin on first miss) vs Push CDN (you upload assets explicitly — better for large video files). stale-while-revalidate is the key cache header pattern: serve cached immediately, refresh in background. Vercel uses this for ISR (Incremental Static Regeneration).
04
Rate Limiter
Caps how many requests a client can make in a time window
routing pm relevant sa relevant
⌄
What it is
A gatekeeper that enforces "you can make at most N requests per time window." When exceeded, returns HTTP 429 Too Many Requests. Can be enforced per user, per IP, per API key, or globally. Lives at the API gateway, in middleware, or in a shared cache like Redis.
What problem it solves
Cost protection: a bug in your code or a malicious user can loop API calls — without a limit, this becomes a huge bill (especially with Claude API calls).
Abuse/DDoS prevention: limits damage from bad actors.
Fairness: one heavy user can't degrade experience for others.
Cascade prevention: stops a slow service from being overwhelmed.
4 algorithms (concept level)
Token bucket: refill N tokens/sec, each request costs 1. Allows bursts. Most common.
Leaky bucket: queue requests, process at fixed rate. Smooth output.
Fixed window: count per minute. Has an edge case at window boundary.
Sliding window: count in last 60s rolling. Fairest, most memory-intensive.
SensAI / Pulse relevance
Pulse queries Claude API per question. Without rate limiting on /api/pulse/query, a loop or bad actor could run thousands of expensive queries. A token bucket (10 queries/min/user) protects cost. For SensAI's Build What Matters cohort: rate limit registration endpoints to prevent spam signups.
When it comes up in conversation
  • Designing pricing tiers → "Free tier: 100 API calls/day. Pro: 10,000/day." Rate limiting is the enforcement mechanism — the PM designs the limits, engineering implements them.
  • Customer hits 429 errors → "You've hit your rate limit. Implement exponential backoff on retries." SA explains this in integration calls.
  • Integrating with Anthropic API → you've seen 429s. That's Anthropic's token bucket. You need retry logic with backoff in your code.
  • Security review: "how do you prevent API abuse?" → Rate limiting + API key authentication is the standard answer.
  • Planning AI feature costs → "We need rate limits or a single power user could spend $500 in a day on LLM calls." Real PM conversation.
SA depth: Rate limits should be communicated via response headers: X-RateLimit-Limit, X-RateLimit-Remaining, Retry-After. Customers integrating your API will ask about these. Distributed rate limiting (multiple servers sharing one counter) requires Redis or a similar shared store — per-server counters get gamed when you have many instances.

Building blocks 05–08

Storage & Data

Where your system's state lives. These are the decisions with the longest-lasting consequences — migrations are expensive, and the wrong choice here constrains your product for years.

⌕
05
Databases (see deep dive tab)
Persistent structured storage — the system's source of truth
storage pm relevant sa relevant deep dive →
⌄
What it is
Software that stores, organizes, and retrieves data reliably. Two main families: SQL (relational — tables with relationships, strict schema, ACID guarantees) and NoSQL (flexible schema, optimized for specific access patterns — documents, key-value, wide-column, graph). See the Database Deep Dive tab for full detail.
When it comes up
  • Feature design: "Can we add flexible custom fields per customer?" → SQL schema migrations are expensive. NoSQL document store might be right here.
  • Scaling conversation: "We're at 10M users, queries are slow." → Read replicas? Sharding? Caching layer? This is an SA technical discovery question.
  • Compliance: "We need GDPR right-to-erasure." → Database design and indexing strategy matters for deletion performance.
  • Enterprise deal: "Can we use our own database?" → Bring-your-own-database is an architectural decision with deep implications.
Go to the "Databases Deep Dive" tab for SQL vs NoSQL decision framework, all NoSQL types, replication, sharding, and the PM/SA conversation guide.
06
Key-Value Store
Ultra-fast lookup: give me the value for this key
storage sa relevant
⌄
What it is
The simplest possible data model: every piece of data has a unique key, you look it up by key. Like a hash map / dictionary, but distributed across many machines. Examples: Redis (in-memory, sub-millisecond), DynamoDB (AWS-managed, infinitely scalable), etcd (configuration, used by Kubernetes).
What problem it solves
Speed. A relational DB query with joins can take 50–200ms. A Redis key lookup takes 0.1ms. Use cases: session storage (user is logged in → session token), feature flags (is_feature_enabled → true/false), rate limit counters, caching DB results, distributed locks.
The tradeoff
You can only look up by the key you put in. No "find all users who signed up last week" — that requires scanning everything. Key-value stores are fast because they give up query flexibility. They complement a database; they don't replace it.
When it comes up
  • "Our API is slow because every request queries the user table." → Cache user sessions in Redis. Read from cache first, fall back to DB.
  • "How do you implement feature flags?" → Redis key per flag, checked on each request. Instant toggle without deploy.
  • Engineer mentions "we're using Redis for this" → You know it's fast, in-memory, and probably ephemeral (Redis can lose data on crash unless configured otherwise).
07
Blob Store
Cheap, durable storage for files, images, videos, backups
storage pm relevant sa relevant
⌄
What it is
Object storage for unstructured data — files, images, videos, PDFs, ML models, database backups. Examples: AWS S3, Google Cloud Storage, Cloudflare R2. You store a file at a key (like a path), retrieve it by that key. Not a filesystem, not a database — designed for large, infrequently-mutated objects.
Why never serve files from app server
An app server handling a 50MB video upload is using a thread that could handle 1,000 API requests instead. And if 100 users download simultaneously, your server is crushed. Blob store + CDN separates the byte-serving problem from the business logic problem. S3 + CloudFront is the standard pattern.
Cost structure (SA-relevant)
Storage: ~$0.02/GB/month (S3 Standard). Cheap.
Egress (downloads): ~$0.09/GB (AWS charges heavily). Use Cloudflare R2 ($0 egress) for cost-sensitive workloads.
Operations: cost per PUT/GET request — matters at massive scale.
When it comes up
  • Any feature with user uploads → "Where do uploaded resumes / portfolios / certificates go?" → Blob store.
  • Video course feature for SensAI → requires blob store + CDN. Cost estimate: storage + CDN egress × expected views.
  • Enterprise: "We need our files in our own S3 bucket." → Bring-your-own-storage architecture. Common enterprise requirement.
  • Compliance: "Can you store files in the EU only?" → S3 bucket region selection. Simple configuration, contractual commitment.
  • "How big is your database?" → Distinguish: structured data (DB) vs blob data (object storage). They scale differently and cost differently.
08
Distributed Cache
In-memory layer that absorbs reads before they hit the database
storage sa relevant
⌄
What it is
A fast in-memory store shared across multiple app servers. When a user requests data, the app checks the cache first. If found (cache hit), returns instantly. If not (cache miss), queries the DB, stores result in cache, returns. Next request gets it from cache. Redis and Memcached are the dominant tools.
Why it matters
DB queries: 10–100ms. Cache hit: 0.1–1ms. A well-implemented cache can serve 90% of reads without touching the database — meaning you need far fewer (and cheaper) DB instances. Facebook's social graph serves billions of reads per day primarily from Memcached.
Eviction & consistency tradeoffs
Eviction policies: LRU (least recently used) — drop oldest items when full. Most common.
Consistency risk: DB updates don't auto-update cache. A user might see stale data for TTL seconds. Fine for dashboards; not fine for balances.
Cache invalidation: delete or update cache when DB changes.
When it comes up
  • "Reports are slow." → Cache the report result for 5 minutes. 99% of loads are repeat views; only 1% need fresh data.
  • Scaling conversation: "We're getting DB read bottlenecks." → Read replicas or caching layer (or both).
  • SensAI: course content, learner progress summaries → cache aggressively. Placement records → short TTL or no cache (must be accurate).
  • Customer asks about "real-time data" → clarify: is real-time truly needed? Cache with 30s TTL often feels real-time and costs 10× less.

Deep dive — building block 05

Databases: SQL vs NoSQL

The most consequential architectural choice in any product. Database migrations are expensive, slow, and risky — which is why this decision deserves more depth than any other building block.

The core difference

SQL databases organize data into tables with defined schemas and relationships. NoSQL databases trade structure and guarantees for speed, flexibility, or scale. Neither is better — the right choice depends entirely on your data's shape and access patterns.

SQL (Relational)
PostgreSQL · MySQL · SQLite · Supabase
Schema: defined upfront. Every row in a table has the same columns.
Relationships: foreign keys, JOINs. Give me all placements for this learner with their company name.
ACID guarantees: Atomicity, Consistency, Isolation, Durability. Transactions either fully complete or fully fail. Critical for financial data.
Query flexibility: SQL lets you ask almost any question of your data — aggregate, filter, join, group by anything.
Scaling: traditionally vertical (bigger server). Horizontal sharding is possible but complex.
NoSQL (Non-relational)
MongoDB · DynamoDB · Cassandra · Redis
Schema: flexible. Each document/record can have different fields. Great for evolving data models.
Relationships: generally avoided. Data is denormalized — you embed related data in one document rather than joining across tables.
Consistency: often eventual consistency (CAP theorem — see earlier module). Faster writes, but reads may be slightly stale.
Query flexibility: limited. You design queries upfront and optimize for specific access patterns. Ad-hoc analysis is harder.
Scaling: built for horizontal scale. Add more nodes, distribute data across them. This is their core advantage.

The decision framework

Ask these questions when evaluating a database choice — for your own products or in customer technical discovery.

Question SQL if… NoSQL if…
Data relationships Data has many relationships. Learner → placements → companies → roles. JOINs are your friend. Data is self-contained. A user's activity feed — each event stands alone, no JOINs needed.
Schema stability Schema is known upfront and changes rarely. Core entities: users, orders, invoices. Schema evolves rapidly. Each customer has different custom fields. Flexible per-document structure needed.
Consistency requirement Must be correct immediately. Bank balances, inventory, placement records. No stale data. Eventual consistency acceptable. Like counts, view counters, activity feeds. Slight lag is fine.
Query patterns Complex ad-hoc queries. Reports, analytics, "show me all X where Y and Z." Need flexibility. Known, repetitive access patterns. "Get user profile by ID." Optimized for that one query at scale.
Scale Moderate scale. Millions of users. Supabase/PostgreSQL handles this fine with read replicas. Massive, unpredictable scale. Billions of records, global distribution, need to add nodes easily.
Transactions Multi-step operations must be atomic. Transfer money: debit A AND credit B, or neither. Each write is independent. Log an event. Record a click. Failure of one doesn't affect others.
PM frame: The database choice constrains your product roadmap. Chose a rigid SQL schema early? Adding flexible custom fields per customer is now a complex migration. Chose an eventual-consistency NoSQL store? Building a financial audit trail on it is now a liability. The right question before any major data model decision is: "what queries will we need to run in 2 years?"

The 4 types of NoSQL

NoSQL is not one thing — it's four different data models optimized for four different problems. Knowing which is which matters when engineers discuss architecture or when you're evaluating a vendor's tech stack.

Document store
MongoDB · Firestore · CouchDB
Stores data as JSON-like documents. Each document can have different fields. Natural fit for objects that vary per record — user profiles, product catalogs, CMS content.
SensAI use: course content, learner profiles with variable fields
Key-value store
Redis · DynamoDB · etcd
Simplest model: key → value. Extremely fast. No query flexibility. Use for caching, sessions, feature flags, rate limit counters, leaderboards.
Pulse use: caching Claude responses, rate limit counters
Wide-column store
Cassandra · HBase · BigTable
Tables with flexible columns per row, optimized for high-write throughput and time-series data. Netflix uses Cassandra for viewing history. Designed for massive write scale.
Use case: IoT sensor data, time-series logs, activity streams at scale
Graph database
Neo4j · Amazon Neptune
Optimized for relationship traversal. "Friends of friends who also liked X." Social networks, recommendation engines, fraud detection (unusual connection patterns), knowledge graphs.
Use case: LinkedIn's social graph, fraud detection networks

Replication & Scaling — what the options mean

When engineers discuss database scaling, these are the three main patterns. As a PM or SA you need to know what each one costs and what it enables.

Read Replicas
Primary handles writes. Copies (replicas) serve reads. Most queries are reads — dashboards, reports, lookups. Replicas lag slightly behind primary (eventual consistency on reads).
When: read-heavy workloads. First scaling move for most apps. Supabase, RDS both support this easily.
Vertical Scaling
Bigger server — more CPU, RAM, faster SSD. Simplest solution. PostgreSQL on a 64-core machine with 512GB RAM handles enormous load. Hard ceiling and single point of failure.
When: your DB is the bottleneck and you haven't maxed out the instance size yet. Cheaper than you think before hitting limits.
Sharding
Split data across multiple databases by a shard key (e.g. by customer ID range). Each shard is a separate DB server. Theoretically infinite scale. Extremely complex — cross-shard queries, rebalancing, operational overhead.
When: you've exhausted read replicas and vertical scaling. Last resort for SQL. Built-in for Cassandra/DynamoDB.
SA conversation: When a customer asks "can you handle our scale?" — the honest progression is: (1) how many reads vs writes? (2) do you need strong consistency? (3) current approach: Postgres with read replicas handles 100K+ RPS for reads. (4) if you're at Cassandra-scale (billions of writes/day), that's a different architecture conversation. Most B2B SaaS never needs to leave Postgres.
PM frame: Multi-tenancy architecture is a database decision. Do you put all customers in one DB (shared schema — simpler, cheaper, harder to isolate), one DB per customer (dedicated — expensive, great isolation, common for enterprise), or one schema per customer in one DB (middle ground)? This directly affects pricing, compliance, and feature velocity. Decide early — migrating later is painful.

For SensAI & Pulse specifically

Concrete recommendations based on your actual systems, not hypotheticals.

SensAI — right now
Supabase (PostgreSQL) is the correct choice and will remain so until tens of thousands of learners. Learner → course → progress → placement is a relational model. SQL gives you the reporting flexibility you need. Row-level security (RLS) in Supabase handles multi-tenant data isolation. You don't need NoSQL.
Pulse — right now
PostgreSQL for the core placement data. Possibly Redis (or even in-memory) for caching frequent query results from Claude. The Pulse golden-set evaluation data (question → expected answer pairs) fits naturally in Postgres. Don't add NoSQL complexity before you need scale.
When to reconsider
If you build a learner activity feed (millions of events/day), consider a time-series or wide-column store. If you add a recommendation engine ("learners like you also studied X"), consider a graph DB or vector DB. If you need per-customer schema flexibility, MongoDB document model becomes attractive. All of these are 10,000+ learner problems.
The vector DB conversation
For AI features — semantic search, similar content, RAG — you need a vector database (pgvector in Postgres, Pinecone, Weaviate). This stores ML embeddings and lets you query by semantic similarity. "Find lessons similar to this one." Supabase has pgvector built in — you already have what you need for basic RAG.

Building blocks 09–10

Async Communication

Decoupling services so they don't need to be running simultaneously. The difference between a brittle system and a resilient one is often whether work is synchronous or async.

09
Messaging Queue
Store work to be done; process it when you're ready
async pm relevant sa relevant
⌄
What it is
A buffer between a producer (sends work) and a consumer (does work). Producer drops a message and moves on — doesn't wait for it to be processed. Consumer picks it up when ready. Examples: AWS SQS, RabbitMQ, Kafka (more powerful, also pub-sub). Slack events to Pulse's backend are effectively a queue.
What problem it solves
Decoupling: if the consumer is down, messages wait — they aren't lost.
Traffic absorption: Uber gets 10K ride requests/sec at New Year's. Queue absorbs the spike; workers process at sustainable rate.
Reliability: if processing fails, messages can be retried automatically.
Long-running work: email sends, ML inference, report generation shouldn't block a web request.
Key concepts
At-least-once delivery: message delivered at least once, possibly more. Consumer must be idempotent (same input → same result).
Dead-letter queue: messages that fail repeatedly go here for inspection.
Ordering: most queues don't guarantee order. Kafka does (within a partition).
When it comes up
  • "Sending welcome emails is slowing down registration." → Put email jobs on a queue. Registration responds immediately, email sends asynchronously.
  • "Our AI processing takes 30 seconds — the user gets a timeout." → Queue the job, return a job ID, poll for completion. Standard async pattern for LLM calls.
  • Reliability question: "What happens if your server crashes mid-processing?" → Unacknowledged queue messages are redelivered. No work lost.
  • SensAI: batch processing learner progress reports, sending placement notifications, running eval pipelines for Pulse → all should be queued, not synchronous.
10
Pub-Sub System
One event, many independent consumers
async sa relevant
⌄
Queue vs Pub-Sub
Queue: one message → one consumer. Work is divided among workers. Process this payment — only one service should do it.
Pub-Sub: one message → all subscribers. Fan-out. User placed order → inventory service receives it AND email service receives it AND analytics receives it. Same event, independent reactions.
What problem it solves
Without pub-sub, adding a new consumer means modifying the producer. With pub-sub, the producer doesn't know or care how many services are listening. New service subscribes, starts receiving events — zero changes to the producer. This is how microservices stay decoupled as systems grow.
When it comes up
  • "When a learner completes a module, we need to update progress, send a certificate, notify the counselor, and update the placement dashboard." → Pub-sub: one event, four independent subscribers.
  • Webhook design: when you send webhooks to customers, you're implementing a pub-sub pattern — your system is the publisher, customer endpoints are subscribers.
  • "How do your integrations work?" → Event-driven architecture via pub-sub means adding a new integration doesn't require changing core product code.
  • Kafka vs SQS: Kafka retains events for days/weeks so late subscribers can replay history. SQS deletes processed messages. Important architectural difference for audit logs and compliance.

Building blocks 11–12

Identity & Counters

Generating unique IDs and handling high-frequency counting — two problems that seem trivial but break in surprising ways at scale.

11
Sequencer / Unique ID Generator
Globally unique IDs that work across distributed nodes
identity sa relevant
⌄
What it is
Every record needs a unique ID. At scale, generating IDs without a central authority is non-trivial — two servers could generate the same ID simultaneously. Three approaches: UUID (random, unique but not sortable, not readable), Snowflake/ULID (timestamp + machine ID + sequence — sortable, unique, readable), Database auto-increment (simple, but DB is a bottleneck).
When it comes up
  • API design review: "Are your IDs sequential integers?" → Sequential IDs are a security risk (user can enumerate records: /users/1, /users/2…). UUIDs or Snowflake IDs prevent this.
  • Data merges: importing learner records from two systems → ID collision risk. Strategy needed before merge.
  • Audit logs: sortable IDs (Snowflake/ULID) mean logs are automatically in chronological order — no ORDER BY needed.
12
Sharded Counters
Counting millions of concurrent events without a write bottleneck
storage sa relevant
⌄
What it is
A naive counter is one DB row — incrementing it from millions of concurrent requests causes a write lock bottleneck. Sharded counters split into N rows (e.g. 100 shards). Each write goes to a random shard. To read the total, sum all shards. Trades write speed for read complexity.
When it comes up
  • Any feature with high-concurrency writes: like counts, view counts, inventory decrements during flash sales.
  • "Why does the like count on a viral post sometimes go up and down?" → Sharded counters + eventual consistency. The count is approximate during high traffic, converges to accurate shortly after.
  • Product decision: do you need exact counts or approximate? Approximate (HyperLogLog algorithm) is 100× cheaper. For view counts, approximate is fine. For inventory, you need exact.

Building blocks 13–14

Search & Compute

Indexing data so it's findable, and executing work outside the request path.

13
Distributed Search
Full-text search and indexing across large datasets
storage pm relevant sa relevant
⌄
What it is
A system that indexes documents and allows fast full-text search. Core data structure: inverted index (word → list of documents containing it). Like a book index, but for your entire dataset. Elasticsearch and Solr (both built on Lucene) are dominant. Postgres has basic full-text search built in (good enough for small scale).
DB query vs search index
DB: WHERE name = 'Gayathri' — exact match, uses B-tree index. Fast.
Search: "find learners who worked in fintech" — full text across multiple fields, fuzzy matching, relevance ranking, synonyms. A SQL LIKE query with % prefix is a full table scan — kills performance at scale. Search indexes are purpose-built for this.
When it comes up
  • Any product with a search bar on user-generated content → Postgres full-text for <100K records, Elasticsearch/Typesense beyond that.
  • Pulse: "Search all placement notes for 'backend role'" → full-text search across free-form notes. Not a SQL query.
  • SensAI course search: "find lessons about system design" → semantic search (vector DB) beats keyword search for educational content.
  • Enterprise: "We need to search across all our documents." → Elasticsearch or OpenSearch (AWS). Cost: indexing pipeline + storage + query compute. Not trivial to operate.
  • Log analysis (Pulse eval pipeline): Elasticsearch is the backbone of the ELK stack — Elasticsearch + Logstash + Kibana. Standard for log search.
14
Distributed Task Scheduler
Run background jobs reliably, on schedule or on trigger
async pm relevant sa relevant
⌄
What it is
A system for running work outside the request-response cycle, either on a schedule (cron jobs — every day at 9am, generate placement report) or triggered by events (user signed up → send onboarding sequence). Examples: Celery (Python), BullMQ (Node), Temporal (complex workflows), GitHub Actions, Railway Cron.
Why it matters
Work that doesn't belong in a web request: ML model retraining, sending daily digests, cleaning up stale data, running evals, generating weekly reports, syncing with external APIs. The scheduler ensures these run reliably, retries on failure, and doesn't block users waiting for a response.
When it comes up
  • Pulse: running the Ask Pulse golden-set eval every night → scheduled task. Slack summary of yesterday's placements → cron job at 8am.
  • SensAI: weekly learner progress emails, certificate generation after course completion, cohort reminders.
  • "Can you send automated follow-ups after 7 days?" → Scheduled task. PM decision: what triggers it, what's the content. Engineering decision: which scheduler, retry behavior.
  • SLA question: "How quickly do you process X?" → Depends on queue depth and worker count. Knowable, tunable — SA can give honest answer with architecture context.

Building blocks 15–16

Observability

How you know what's happening inside a running system. Often the last thing built, always the first thing missed during an incident.

15
Distributed Logging
Collect and search logs from every service in one place
ops sa relevant
⌄
What it is
A pipeline that aggregates logs from all services into a central searchable store. Without it, debugging means SSHing into each server and grepping log files — impossible when you have 50 services. Standard stack: Fluentd/Logstash (collect) → Kafka (buffer) → Elasticsearch (store/index) → Kibana (visualize). Or managed: Datadog, Grafana Cloud, AWS CloudWatch.
Structured vs unstructured logs
Unstructured: "User 123 logged in at 14:32" — readable but hard to query.
Structured (JSON): {"user_id": 123, "event": "login", "timestamp": "14:32", "ip": "…"} — queryable, filterable, aggregatable. Always use structured logs for production systems.
When it comes up
  • Security review: "Do you have audit logs?" → All user actions logged with timestamps and actor IDs, searchable, retained for N years. Compliance requirement.
  • Incident: "What happened at 3pm yesterday?" → Centralized logs with millisecond timestamps across all services. SA can speak to log retention policy and search capability.
  • Pulse eval pipeline: logging each query, Claude's response, the eval score → structured logs that feed your golden-set dashboard.
  • Enterprise procurement: "Where are logs stored? Who can access them? How long are they retained?" → Standard security questionnaire item. SA owns this answer.

Service Monitoring
Know your system is healthy before your users tell you it isn't
ops pm relevant sa relevant
⌄
What it is
Collecting and alerting on metrics about system behavior. The four golden signals (Google SRE): Latency (how long requests take), Traffic (requests per second), Errors (error rate), Saturation (how full is your resource — CPU, memory, disk). Tools: Datadog, Prometheus + Grafana, New Relic, Sentry (error tracking).
SLOs, SLIs, SLAs
SLI (indicator): measured metric — e.g. % of requests under 200ms.
SLO (objective): internal target — e.g. 99.9% of requests under 200ms.
SLA (agreement): contractual commitment to customers — e.g. 99.9% uptime or credit issued.
SLAs should be looser than SLOs — you commit to customers what you're confident achieving internally.
When it comes up
  • Enterprise procurement: "What's your uptime SLA?" → 99.9% (8.7hrs downtime/yr) is standard. 99.99% requires significant investment. SA needs to know what's contractually offered and what the monitoring/alerting infrastructure actually supports.
  • Customer escalation: "Your product was slow yesterday." → Monitoring tells you exactly when, for how long, which endpoints, which users. Evidence-based incident response vs guessing.
  • PM roadmap: "Should we invest in monitoring?" → Monitoring isn't optional above a certain scale. Invisible failures = lost customers. MTTR (Mean Time To Recovery) is a product metric.
  • Pulse: monitoring Claude API latency, error rates on queries, Slack webhook delivery success — these are your product health metrics. Sentry for error tracking is a 30-minute setup that pays dividends immediately.
SA depth: Distinguish between monitoring (are we healthy?) and observability (why are we unhealthy?). Observability = logs + metrics + traces. Distributed tracing (Jaeger, Datadog APM) lets you follow a single request across 10 microservices and see exactly where it slowed down. Critical for complex architectures.
