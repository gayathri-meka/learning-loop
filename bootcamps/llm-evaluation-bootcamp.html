Day 01 — Foundations

The eval mindset

Before any tooling: why evaluating an LLM is unlike testing normal software, and the small set of ideas that make quality measurable at all.

The problem with “it looks good”

Normal software is deterministic: same input, same output, and a test that passed yesterday passes today. LLM systems break both halves of that promise. The same prompt can return different text on each call, and a model or prompt change that fixes one case can silently regress ten others you weren't watching. “I tried a few examples and they looked good” is how every team starts — and it stops working the moment you have more than a handful of cases or more than one person editing prompts.

There's a second trap: quality is not one number. An answer can be factually correct but rude, fluent but unfaithful to its sources, helpful but leaking PII. A single “goodness” score hides exactly the failure you care about. The job of evaluation is to make these dimensions explicit and countable.

Mental model

Evals are unit tests for non-deterministic systems. You can't assert output == expected — so you assert that it scores above a bar, on dimensions you chose in advance.

The shape of every eval

Strip away the frameworks and every evaluation is the same five-step loop. Internalize it once and every tool you touch becomes a variation on this.

  ┌──────────────┐     ┌──────────┐     ┌───────────┐     ┌────────────┐     ┌──────────┐
  │  1. DATASET  │ ──▶ │  2. RUN  │ ──▶ │ 3. SCORE  │ ──▶ │ 4.AGGREGATE│ ──▶ │ 5.DECIDE │
  └──────────────┘     └──────────┘     └───────────┘     └────────────┘     └──────────┘
   golden inputs        your system      a scorer per      mean / pass-rate    ship? block?
   (+ refs/rubrics)      under test       quality dim       per dimension       investigate?
       ▲                                                                          │
       └──────────────────────  failures feed back into the dataset  ◀───────────────┘
The eval loop. Step 3 is where the design lives — the rest is plumbing.

That loop runs in three different places, and conflating them is a classic mistake. Offline evals run against a curated dataset on your laptop or in a notebook — fast iteration. CI evals run the same suite automatically before a prompt or model change merges — the gate. Online evals run against a sample of real production traffic — catching drift the curated set never anticipated. You want all three; each catches what the others miss.

Three kinds of scorer

Step 3 — scoring — comes in exactly three flavors, in increasing order of power and cost. Most of the craft of evaluation is knowing which to reach for.

1

Deterministic

Code-based checks: exact match, “contains”, regex, JSON-schema validity, latency under N ms. Cheap, fast, zero ambiguity — but only works when there's a checkable right answer.

2

Model-graded (rubric)

An LLM judge scores against criteria you define — correctness, faithfulness, tone. Handles free text where there's no single right answer. The whole of Day 2.

3

Human

The ground truth everything else is calibrated against. Expensive and slow, so you spend it deliberately — usually to label a few hundred examples that anchor your automated scorers.

★

Leaderboards ≠ your app

Public benchmarks (MMLU, LMArena) measure a bare model with a generic prompt. They tell you almost nothing about your system on your data. Build your own eval set.

Build · a first eval harness

Start where every eval should: a tiny golden dataset and deterministic scorers. We'll evaluate Pulse on questions that do have checkable answers.

eval_pulse_v0.py
# A golden dataset: inputs + what a correct answer must contain.
golden = [
  {"q": "Which candidates are placed at Acme?",
   "must_contain": ["Priya", "Marcus"], "must_be_json": True},
  {"q": "How many open roles at Byte Corp?",
   "must_contain": ["3"], "must_be_json": False},
]

def score(case, output):
  results = {}
  # Deterministic scorer 1: required facts present
  results["contains"] = all(s in output for s in case["must_contain"])
  # Deterministic scorer 2: structural validity
  if case["must_be_json"]:
    results["valid_json"] = is_valid_json(output)
  return results

# Run + aggregate
rows = [score(c, pulse(c["q"])) for c in golden]
pass_rate = mean([all(r.values()) for r in rows])
print(f"pass rate: {pass_rate:.0%}")  # your first defensible number
What to observe

Run it against a free-text question — “summarize Priya's placement history” — and watch must_contain fail on a perfectly good answer that happens to phrase things differently. That brittleness is the wall every team hits, and it's exactly what Day 2 exists to solve.

Day 02 — Automation

The judge

When there's no single right answer, you grade with another model. This is the workhorse of modern evaluation — and the one most teams get subtly wrong.

Letting a model grade a model

Day 1's deterministic scorers fall apart on free text — a SensAI tutoring explanation has a thousand valid phrasings. The breakthrough idea, now standard, is LLM-as-judge: prompt a strong model with your criteria and have it score the output. The bet is empirical and it mostly holds — on many tasks a well-built judge agrees with human reviewers roughly 85% of the time, which is often higher than two humans agree with each other. That's good enough to run at a scale no human panel could touch.

But a judge is a model too. It is non-deterministic, it can be wrong, and — critically — it has systematic biases that inflate or distort your numbers if you don't design around them. The single biggest mistake in evaluation is trusting a judge you never validated.

Mental model

The judge is part of your system under test. Before it grades your app, you have to grade the judge — against humans, on your data.

Three ways to ask the question

How you frame the judging task changes its reliability. There are three standard modes.

  SINGLE-ANSWER GRADING        PAIRWISE COMPARISON          REFERENCE-BASED
  ┌───────────────────┐        ┌───────────────────┐        ┌───────────────────┐
  │  output + rubric  │        │  output A vs B    │        │  output + gold ref│
  │        ▼          │        │        ▼          │        │        ▼          │
  │   score 1–5       │        │  which is better? │        │  matches ref?     │
  └───────────────────┘        └───────────────────┘        └───────────────────┘
  absolute, scalable;          most reliable for           strongest signal, but
  scores can drift             "did B regress?"            needs a gold answer
Reference-free (left two) scales without gold answers; reference-based is most precise when you have them.

The judge's biases — and the fixes

These aren't edge cases; they're measured, repeatable, and large enough to flip conclusions. Know them by name.

1

Position bias

In pairwise mode, judges favor whichever answer is shown first — often ~60% of the time. Swapping the two can flip the verdict in ~40% of pairs. Fix: run both orders, average.

2

Verbosity / length bias

Longer answers get scored higher regardless of quality. Fix: say so in the rubric, cap length, or normalize for it.

3

Self-preference

A judge tends to favor outputs from its own model family. Fix: judge with a different model than the one under test where you can.

4

Sycophancy / agreeableness

Judges over-accept plausible-looking output without scrutiny. Fix: force chain-of-thought reasoning before the score, and demand evidence.

Two practices neutralize most of this. First, make the judge reason before it scores (the “G-Eval” pattern): a rubric plus step-by-step justification beats a bare number. Second, calibrate against humans: hand-label at least ~100 examples, then check that your judge correlates with them before you trust a single automated run. An uncalibrated judge is a confident random number generator.

Build · an LLM judge with a rubric

Grade SensAI tutoring answers on two dimensions, with reasoning-before-score and a position-swap helper for pairwise mode.

judge_sensai.py
# Reasoning-before-score rubric (the G-Eval pattern)
JUDGE_PROMPT = """You are grading a tutoring answer for a learner.
Criteria:
  - correctness: is the explanation factually right?
  - clarity: would a beginner follow it?
First reason step by step about each criterion.
THEN output JSON: {"correctness": 1-5, "clarity": 1-5, "why": "..."}"""

def judge(question, answer, judge_model="a-different-model"):
  msg = f"{JUDGE_PROMPT}\n\nQ: {question}\nANSWER: {answer}"
  return parse_json(llm(judge_model, msg))   # {"correctness":4,"clarity":5,...}

# Pairwise with position-swap to cancel position bias
def pairwise(q, a, b):
  v1 = judge_pair(q, a, b)        # A first
  v2 = judge_pair(q, b, a)        # B first
  return "tie" if v1 != flip(v2) else v1   # disagreement on swap = tie

# ALWAYS validate the judge before trusting it
agreement = correlate(judge_all(human_labeled), human_scores)
assert agreement > 0.8, "judge not calibrated — do not ship on this number"
What to observe

Run pairwise without the swap and you'll see a confident winner. Add the swap and a chunk of those “wins” turn into ties — that's position bias you were about to ship as signal. Then feed the judge a padded, waffly answer and watch verbosity bias inflate the score.

Day 03 — Production

The gate

An eval that lives in a notebook changes nothing. The payoff is an eval that runs in CI, blocks a bad deploy, and watches live traffic — plus the metrics specific to RAG and agents.

Eval-driven development

The mindset shift that makes all of this pay off: write the eval before the feature. The eval set is the spec — it states, in checkable form, what “working” means. Then the same suite runs automatically in CI, and a prompt or model change that drops a metric below its bar simply doesn't merge. Quality stops being a vibe and becomes a gate, the same way tests gate normal code.

Mental model

Ship the eval before the feature. If you can't write a test that tells you it's working, you don't yet know what “working” means.

   author change          CI GATE                       PRODUCTION
   ┌────────────┐      ┌──────────────┐               ┌──────────────────┐
   │ new prompt │ ───▶ │ run eval     │ ── pass ────▶ │ deploy           │
   │ or model   │      │ suite        │               │      │           │
   └────────────┘      └──────┬───────┘               │      ▼           │
                              │ fail                    │ online eval on   │
                              ▼                         │ sampled traffic  │
                       block the merge             └────────┬─────────┘
                                                            │ surprising failures
       └──────────────────  add to golden dataset  ◀───────────┘
Offline iterate → CI gate → online watch → failures feed the dataset. The loop closes.

Metrics that fit the system

Generic “is it good” isn't enough once you're evaluating RAG or agents. Each has purpose-built metrics that pinpoint where it broke.

R

RAG · faithfulness

Is every claim in the answer supported by the retrieved context? Catches hallucination even when the answer “sounds right.” The headline RAG metric for Pulse.

R

RAG · context precision/recall

Did retrieval surface the right chunks (recall) without burying them in junk (precision)? Separates a retrieval bug from a generation bug.

A

Agent · task success + trajectory

Did it reach the goal (success), and take a sound path of tool calls to get there (trajectory accuracy)? A right answer via a broken path is a latent failure.

A

Agent · pass^k consistency

Run the same task k times. pass^1 is “can it”; pass^k is “can it reliably.” The gap is often 15–25 points — and it's what production actually feels.

One discipline underpins all of it: guard the eval set itself. If your eval examples leak into training or prompt context, your numbers inflate — this is benchmark contamination, and it's why public leaderboard scores are widely inflated by 5–15 points. Keep a held-out set, version it, and report a metric with a confidence interval, not a single hero number.

Build · an eval suite that gates the deploy

A pytest-style suite for Pulse that asserts faithfulness and consistency. Wire it into CI and a failing metric blocks the merge.

test_pulse_eval.py
# Runs in CI on every prompt/model change. Failing assert = blocked merge.
THRESHOLDS = {"faithfulness": 0.90, "context_recall": 0.85}

def test_pulse_faithfulness():
  scores = []
  for case in golden_rag:
    answer, context = pulse_with_context(case["q"])
    # model-graded: every claim must be supported by context
    scores.append(faithfulness(answer, context))
  assert mean(scores) >= THRESHOLDS["faithfulness"], \
    f"faithfulness {mean(scores):.2f} below bar — fix before merge"

def test_pulse_consistency():       # pass^k: capability vs reliability
  task = "book a screening for Priya at Acme"
  oks = [agent_succeeds(task) for _ in range(5)]
  assert all(oks), f"pass^5 failed: {sum(oks)}/5 — flaky in prod"
What to observe

Write an answer that's fluent and wrong — confidently citing a placement that isn't in the context — and watch faithfulness catch what a generic “quality” score would have waved through. Then watch test_pulse_consistency go red even though the task “works” when you try it once by hand.

Interview prep

Say it like someone who's shipped evals

The questions that separate “read a blog post” from “ran this in production.” Answer with the vocabulary, the trade-off, and a concrete example.

How would you evaluate an LLM feature where there's no single right answer?
Move up the scorer ladder. Deterministic where I can (format, required facts), then an LLM-as-judge with a rubric for the open-ended quality dimensions, calibrated against ~100 human labels first. I'd score multiple dimensions separately — a single number hides the failure that matters — and report pass-rates with a confidence interval, not one hero score.
Your LLM judge says version B is better. Why might you not trust it?
Judges have measurable biases. Position bias — it may just prefer whichever was shown first — so I'd swap order and re-run; if the verdict flips, it's a tie. Plus verbosity and self-preference bias. And I'd ask whether the judge was ever calibrated against humans on this task. Until it is, it's a confident guess.
A model scores 90% on a benchmark. What does that tell you about production?
Less than it looks. Public scores are inflated ~5–15 points by contamination and single-run reporting, and they measure a bare model — not my system on my data. pass^1 isn't pass^k: 90% capability can be ~70% reliability when real users retry. I'd trust my own held-out eval over any leaderboard.
What does “eval-driven development” mean in practice?
The eval is the spec, written first. The suite runs in CI as a gate — a prompt or model change that drops a metric below its bar doesn't merge. Then online eval on sampled traffic catches drift, and those surprises feed back into the golden dataset. Quality becomes a gate, not a vibe.
For a RAG system like ours, which metric do you watch most?
Faithfulness — is every claim supported by retrieved context — because it catches confident hallucination that a generic quality score waves through. I'd pair it with context precision/recall to localize the bug: low recall is a retrieval problem, high recall but low faithfulness is a generation problem.
Quick reference

The one-page cheat sheet

The three scorer types

  • Deterministic — exact match, contains, regex, JSON-valid, latency. Cheap, zero ambiguity.
  • Model-graded — LLM judge + rubric. For free text; calibrate it first.
  • Human — the ground truth; spend it to calibrate the other two.

Where evals run

  • Offline — curated dataset, fast iteration.
  • CI — same suite, gates the merge.
  • Online — sampled prod traffic, catches drift.

Judge bias checklist

  • Position — swap order, average.
  • Verbosity — penalize length in the rubric.
  • Self-preference — judge with a different model.
  • Sycophancy — reason before scoring; demand evidence.

System-specific metrics

  • RAG — faithfulness, context precision/recall, answer relevancy.
  • Agent — task success, trajectory accuracy, pass^k.
  • Always — multidimensional, CI with confidence intervals.
