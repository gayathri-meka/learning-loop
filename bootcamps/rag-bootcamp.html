Day 01 Foundations

1.1 What is RAG and Why Does It Exist?

An LLM is a generalist. It was trained on internet text up to a cutoff date. It knows a lot about the world, but it doesn't know your documents, recent events, or private domain-specific data.

The naive solution: put everything in the prompt. The problem: context windows are finite. You can't paste 10,000 pages into a prompt. And even if you could, the model buries the needle in the haystack.

RAG (Retrieval-Augmented Generation) splits the problem: before the query, process and index your documents. At query time, retrieve only the relevant parts, then generate an answer.

Mental model

Instead of handing a student the entire library before asking a question, you give them a librarian who pulls the right 3 pages.

1.2 The Pipeline Anatomy

Every RAG system has two phases:

INGESTION PHASE (offline, run once) Raw Documents ↓ Load ↓ Chunk ↓ Embed ↓ Vector Store QUERY PHASE (online, every request) User Query ↓ Embed Query ↓ Retrieve (similarity search) ↓ Stuff into Prompt ↓ Generate Answer ↓ Response

1.3 Key Concepts

Embeddings

An embedding is a list of numbers (a vector) that represents the meaning of a piece of text.

"The dog ran fast"        → [0.2, -0.8, 0.4, ...]
"The canine sprinted quickly" → [0.21, -0.79, 0.41, ...]

These sentences have similar vectors because they mean similar things. OpenAI's text-embedding-3-small produces 1536-dimensional vectors. Key idea: similar meaning = similar vector = close in space.

Vector Stores

A database optimized for storing and searching vectors. Instead of WHERE name = 'John', you ask "give me the 5 vectors closest to this query vector." We'll use ChromaDB — runs locally, no setup needed.

Cosine Similarity

The metric used to measure how "close" two vectors are. Score of 1 = identical meaning. Score of 0 = unrelated. ChromaDB handles this automatically.

Chunking

You can't embed a 50-page document as one vector. You split it into chunks (~500 characters each) with some overlap so context isn't lost at boundaries.

1.4 Day 1 Build: Ingest a Document

Save this as day1.py:

import os
from dotenv import load_dotenv
from langchain_community.document_loaders import PyPDFLoader
from langchain.text_splitter import RecursiveCharacterTextSplitter
from langchain_openai import OpenAIEmbeddings
from langchain_community.vectorstores import Chroma

load_dotenv()

# --- Step 1: Load your document ---
loader = PyPDFLoader("data/your_document.pdf")
documents = loader.load()
print(f"Loaded {len(documents)} pages")

# --- Step 2: Chunk it ---
splitter = RecursiveCharacterTextSplitter(
    chunk_size=500,
    chunk_overlap=50
)
chunks = splitter.split_documents(documents)
print(f"Split into {len(chunks)} chunks")
print(chunks[0].page_content)

# --- Step 3: Embed and store ---
embeddings = OpenAIEmbeddings(model="text-embedding-3-small")
vectorstore = Chroma.from_documents(
    documents=chunks,
    embedding=embeddings,
    persist_directory="./chroma_db"
)
print("Vector store created and saved.")

# --- Step 4: Sanity check ---
query = "What is this document about?"
results = vectorstore.similarity_search(query, k=3)
for i, doc in enumerate(results):
    print(f"[{i+1}] {doc.page_content[:200]}...")

What to observe

How many chunks your document produced. Whether the top 3 results seem relevant. The sample chunk — does the splitting make sense?

Day 02 Making It Work

2.1 Why Naive RAG Fails

You now have a working retrieval system. But naive RAG breaks in predictable ways:

#ProblemWhat happens
1Bad chunkingChunk cuts mid-concept → noisy embedding → wrong retrieval
2Vocabulary mismatchUser says "remote work," doc says "work from home" → miss
3Top-k noise5 retrieved chunks, 3 are wrong → model hallucinates
4Lost in the middleLLMs focus on start/end → relevant chunk 6 of 10 gets missed

2.2 Reranking

After retrieving top-k chunks, a reranker scores each against the query more precisely and reorders them. Slower but more accurate.

Retrieve top 10 (fast, approximate) ↓ Rerank top 10 → get true top 3 (slower, precise) ↓ Pass top 3 to LLM

2.3 Query Decomposition

Some questions are compound: "What did the company earn in Q3, and how does that compare to Q2?" — this is two questions. Naive RAG treats it as one and retrieves a mixed bag. Query decomposition splits it into sub-queries, retrieves for each, then synthesizes.

2.4 Evaluation: How Do You Know Your RAG Is Good?

Anyone can build a RAG pipeline. Knowing if it's working is the hard part — and the thing Jubin's team actually cares about.

MetricQuestion it answersHow
Retrieval precisionAre the retrieved chunks relevant?Human review or LLM-as-judge
Answer faithfulnessDoes the answer stick to retrieved context?LLM-as-judge
Answer relevanceDoes the answer address the question?LLM-as-judge

2.5 Day 2 Build: Full Pipeline with Eval

Save as day2.py:

import os
from dotenv import load_dotenv
from langchain_community.vectorstores import Chroma
from langchain_openai import OpenAIEmbeddings, ChatOpenAI
from langchain.chains import RetrievalQA
from langchain.prompts import PromptTemplate

load_dotenv()

# --- Load existing vector store ---
embeddings = OpenAIEmbeddings(model="text-embedding-3-small")
vectorstore = Chroma(
    persist_directory="./chroma_db",
    embedding_function=embeddings
)
retriever = vectorstore.as_retriever(
    search_type="similarity",
    search_kwargs={"k": 4}
)

# --- Custom prompt (forces grounding) ---
template = """You are a helpful assistant. Use ONLY the context below to answer.
If the answer is not in the context, say "I don't have enough information."

Context:
{context}

Question: {question}
Answer:"""

prompt = PromptTemplate(
    template=template,
    input_variables=["context", "question"]
)

llm = ChatOpenAI(model="gpt-4o-mini", temperature=0)
qa_chain = RetrievalQA.from_chain_type(
    llm=llm, chain_type="stuff", retriever=retriever,
    chain_type_kwargs={"prompt": prompt},
    return_source_documents=True
)

def ask(question):
    result = qa_chain.invoke({"query": question})
    print(f"A: {result['result']}")
    return result

result = ask("YOUR QUESTION HERE")

# --- LLM-as-judge eval ---
def evaluate_answer(question, context, answer):
    judge = ChatOpenAI(model="gpt-4o-mini", temperature=0)
    prompt = f"""Score this RAG answer:
1. Faithfulness (1-5): Does it only use info from context?
2. Relevance (1-5): Does it address the question?

Question: {question}
Context: {context}
Answer: {answer}

Format: Faithfulness: X/5 - reason\nRelevance: X/5 - reason"""
    return judge.invoke(prompt).content

ctx = "\n".join([d.page_content for d in result['source_documents']])
print(evaluate_answer("YOUR QUESTION", ctx, result['result']))

Day 03 Architecture + Production

3.1 Agentic RAG

Classic RAG is a fixed pipeline: query → retrieve → generate. The LLM has no control. Agentic RAG gives the LLM agency over the retrieval process:

User Query ↓ Agent decides: do I need to retrieve? what query to use? ↓ Retrieves (possibly multiple times) ↓ Agent decides: is this enough? should I refine the query? ↓ Generates final answer

The agent can decide whether retrieval is needed, rewrite the query if the first retrieval fails, retrieve from multiple sources, or use tools beyond retrieval. This is where the field is moving — and why Jubin's team needs an FDE who can wire this correctly for each client.

3.2 Architecture Tradeoffs

When proposing an architecture to a client, these are the decisions you're making:

DecisionOptionsTradeoff
Chunk sizeSmall (200) vs Large (1000 tokens)Small = precise, may lose context. Large = more context, noisier
RetrievalDense vs Sparse (BM25) vs HybridDense = semantic. Sparse = exact keyword. Hybrid = best of both
Top-k3 vs 10More context but more noise and higher cost
RerankingYes vs NoBetter precision, adds latency and cost
Embedding modelOpenAI vs OSS (BGE, E5)OpenAI = easy, costs per call. OSS = free, requires hosting
Generation modelGPT-4o vs GPT-4o-miniBetter quality vs lower cost/latency

Key insight

For precision-sensitive domains like research or legal, bias toward smaller chunks, hybrid retrieval, and reranking even at higher cost. For high-volume consumer use cases, optimize for latency and cost first.

3.3 The Components Your Friend Mentioned

Query Decomposer

Breaks compound questions into sub-queries. Important when documents are structured around specific topics and a single query might span multiple.

Re-ranker

A cross-encoder model that re-scores retrieved chunks against the query. Common choices: Cohere Rerank API, or open source cross-encoder/ms-marco-MiniLM-L-6-v2 from HuggingFace.

Retrieval Types

  • Dense: Vector similarity (what we built)
  • Sparse: BM25 — like a smart grep with TF-IDF scoring
  • Hybrid: Run both, merge results. Best for most production cases

Metadata Filtering

Filter by metadata (date, document type, department) before vector search. Narrows the search space and improves precision — often the biggest win in real deployments.

3.4 Day 3 Build: Agentic Pipeline (Your Demo)

Save as day3.py. This is your demo artifact:

import os
from dotenv import load_dotenv
from langchain_community.vectorstores import Chroma
from langchain_openai import OpenAIEmbeddings, ChatOpenAI
from langchain.agents import AgentExecutor, create_react_agent
from langchain.tools.retriever import create_retriever_tool
from langchain import hub

load_dotenv()

embeddings = OpenAIEmbeddings(model="text-embedding-3-small")
vectorstore = Chroma(
    persist_directory="./chroma_db",
    embedding_function=embeddings
)
retriever = vectorstore.as_retriever(search_kwargs={"k": 4})

# --- Wrap retriever as a tool the agent can call ---
retriever_tool = create_retriever_tool(
    retriever,
    name="document_search",
    description="Search the document knowledge base. Use when answering questions about the documents."
)

llm = ChatOpenAI(model="gpt-4o-mini", temperature=0)
prompt = hub.pull("hwchase17/react")

agent = create_react_agent(llm, [retriever_tool], prompt)
agent_executor = AgentExecutor(
    agent=agent,
    tools=[retriever_tool],
    verbose=True,  # shows agent reasoning — keep this on for the demo
    handle_parsing_errors=True
)

print("RAG Agent ready. Type 'quit' to exit.\n")
while True:
    question = input("You: ").strip()
    if question.lower() == 'quit': break
    if not question: continue
    result = agent_executor.invoke({"input": question})
    print(f"\nAgent: {result['output']}\n")
    print("-" * 50)

Demo tip

verbose=True shows the agent's reasoning chain. This is your demo narrative: "here you can see the agent deciding to search, running the query, evaluating the result, then generating." Leave it on.

3.5 What to Say in the Interview

"The hardest part isn't generation — it's retrieval precision. You can have the best LLM in the world but if you're retrieving noisy chunks, the answer degrades. Most of the engineering work is in chunking strategy, embedding model selection, and hybrid retrieval."

On the retrieval problem

"Classic RAG is a fixed pipeline. Agentic RAG gives the model control over when and how to retrieve — it can decompose queries, retry with refined queries, or decide retrieval isn't needed at all. That's where most production systems are heading."

On agentic RAG

"You can't improve what you don't measure. We need faithfulness — does the answer stay grounded in retrieved context — and relevance — does it actually answer the question. LLM-as-judge is a practical way to run evals at scale."

On evaluation

"The tradeoffs depend entirely on the use case. For precision-sensitive domains like research or legal, I'd bias toward smaller chunks, hybrid retrieval, and reranking even at higher cost. For high-volume consumer use cases, I'd optimize for latency and cost first."
