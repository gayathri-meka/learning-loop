Day 01 Foundations

1.1 What Is an Agent, and Why Do They Exist?

An LLM on its own is a text-in, text-out function. It can't check today's weather, run a query, send an email, or notice that its own last answer was wrong and try again. It predicts tokens and stops.

An agent wraps that model in a loop and hands it tools. The model is given a goal, decides on an action, the action runs, the result comes back, and the model decides what to do next — until it judges the task done. The atomic unit is sometimes called the augmented LLM: a model plus tools, memory, and the ability to retry.

Workflow vs. agent — the distinction that matters

Anthropic's framing is the cleanest in the field:

  • Workflows are systems where LLMs and tools are orchestrated through predefined code paths. You wrote the control flow.
  • Agents are systems where the LLM dynamically directs its own process — deciding which tools to use and when, in a loop you didn't hardcode.

Most production "AI features" are workflows, not agents. That's not a failure — it's usually the right call. Agents trade predictability and cost for flexibility.

Mental model

A workflow is a recipe — fixed steps in a fixed order. An agent is a cook who tastes the dish and decides what to do next. Use a recipe when you can write the steps down. Hire a cook when you can't.

1.2 The Agent Loop

Every agent, no matter the framework, is the same loop underneath:

THE AGENT LOOP Goal / Task ↓ ┌──────────────┐ │ 1. REASON │ model thinks: what do I need to do? └──────┬───────┘ ↓ ┌──────────────┐ │ 2. ACT │ model emits a tool call └──────┬───────┘ ↓ ┌──────────────┐ │ 3. OBSERVE │ tool runs, result returns to the model └──────┬───────┘ ↓ done? ──no──┐ │ │ yes └──► back to 1. REASON ↓ Answer

This is the ReAct pattern (Reason + Act), from the 2022 paper of the same name: interleave reasoning traces with actions so the model can plan, act, and adjust based on what it observes. Nearly every agent framework is a more ergonomic wrapper around this loop.

1.3 Agentic Design Patterns

There are two families of patterns. Knowing which one you need is half the job.

Workflow patterns — you own the control flow

Deterministic scaffolding around LLM calls. From Anthropic's Building Effective Agents:

PatternWhat it doesUse when
Prompt chainingOutput of one LLM call feeds the next, in sequenceTask decomposes into fixed steps
RoutingClassify the input, send it to a specialized pathDistinct categories need distinct handling
ParallelizationRun subtasks at once, aggregate resultsIndependent subtasks, or voting for reliability
Orchestrator-workersA lead LLM splits work, delegates to workers, synthesizesSubtasks aren't known until runtime
Evaluator-optimizerOne LLM generates, another critiques, loop until goodYou have clear eval criteria and iteration helps

Agentic capability patterns — the model owns the control flow

Andrew Ng's framing of what makes an agent capable:

  • Reflection: the model critiques its own output and revises it.
  • Tool use: the model calls external functions to act on the world.
  • Planning: the model breaks a goal into sub-steps and sequences them.
  • Multi-agent collaboration: multiple specialized agents divide labor.

The one rule

Start with the simplest thing that works. A single well-prompted LLM call beats a workflow; a workflow beats an agent; one agent beats many. Add complexity only when a simpler design demonstrably fails. The most common mistake in this field is reaching for a multi-agent swarm to solve a routing problem.

1.4 Day 1 Build: A Bare-Metal Agent (No Framework)

Before any framework, build the loop by hand. This is the single most clarifying thing you can do — every abstraction you meet later is sugar on top of this. Save as day1_agent.py:

import json
from dotenv import load_dotenv
from openai import OpenAI

load_dotenv()
client = OpenAI()

# --- 1. Define a tool (just a Python function) ---
def get_weather(city: str) -> str:
    # A real tool would call an API. We fake it.
    return json.dumps({"city": city, "temp_c": 21, "sky": "clear"})

tools = [{
    "type": "function",
    "function": {
        "name": "get_weather",
        "description": "Get current weather for a city.",
        "parameters": {
            "type": "object",
            "properties": {"city": {"type": "string"}},
            "required": ["city"]
        }
    }
}]

# --- 2. The agent loop ---
def run_agent(user_input):
    messages = [{"role": "user", "content": user_input}]

    while True:
        resp = client.chat.completions.create(
            model="gpt-4o-mini",
            messages=messages,
            tools=tools
        )
        msg = resp.choices[0].message
        messages.append(msg)

        # No tool call → the model is done. Return its answer.
        if not msg.tool_calls:
            return msg.content

        # Otherwise, run each tool call and feed results back.
        for call in msg.tool_calls:
            args = json.loads(call.function.arguments)
            result = get_weather(**args)
            print(f"  [tool] get_weather({args}) -> {result}")
            messages.append({
                "role": "tool",
                "tool_call_id": call.id,
                "content": result
            })

print(run_agent("What should I wear in Lisbon today?"))

What to observe

The model decides on its own to call get_weather — you never told it to. The loop runs until the model stops emitting tool calls. That while True with a "no tool calls → done" exit is the agent. Everything in Days 2 and 3 is about making this loop stateful, controllable, and connectable.

Day 02 Orchestration

2.1 Why the Bare-Metal Loop Breaks

Your Day 1 agent works for one-shot tasks. Push it toward anything real and it cracks in predictable ways:

#ProblemWhat happens
1No persistent stateEvery run starts blank — no memory across turns or sessions
2No human-in-the-loopCan't pause for approval before a risky action (send, pay, delete)
3Rigid control flowBranching, retries, and cycles become a tangle of if-statements
4No resumeCrash at step 7 of 10 → start over from zero
5OpaqueDebugging means print statements; no view of state over time

2.2 LangChain vs. LangGraph

These get conflated constantly. They solve different problems and live at different layers.

LangChain ── the toolkit ────────────────────────── model wrappers · tool abstractions · LCEL chains prebuilt components · loaders · output parsers best for: linear chains, "do A then B then C" LangGraph ── the orchestration layer ────────────── models the agent as a GRAPH: nodes (work) · edges (transitions) · state (memory) built for: cycles, branching, human-in-the-loop, persistence, multi-agent best for: anything stateful or non-linear

Rule of thumb: if you can draw your app as a straight line, LangChain's chains are enough. The moment you need a loop, a decision, or memory that survives, you want LangGraph. LangGraph reached a stable v1.0 in late 2025, so its core API is no longer a moving target.

2.3 LangGraph Core Concepts

State

A shared data structure (a TypedDict or Pydantic model) that every node reads and writes. This is the agent's working memory, passed along the graph. For chat agents it usually holds a growing list of messages.

Nodes

Units of work. A node is just a function that takes the state and returns an update to it. A node can be a plain function, an LLM call, or a whole sub-agent.

Edges & conditional edges

Edges are fixed transitions: "after node A, always go to B." Conditional edges route based on state: "after the model node, if there's a tool call go to the tool node, else finish." Conditional edges are how you express the agent loop as a graph.

Checkpointing

Attach a checkpointer (e.g. InMemorySaver, or a database for production) and the graph saves its state at every step. This single feature unlocks memory (resume a thread by id), time-travel (replay from any step), and fault tolerance (resume after a crash).

Human-in-the-loop

Because state is checkpointed, you can interrupt the graph before or after a node, surface it to a human for approval or edits, then resume exactly where it paused. This is the feature that makes agents safe to put in front of real users.

create_react_agent — the shortcut

For a standard tool-calling agent you don't build the graph by hand. create_react_agent (from langgraph.prebuilt) builds the StateGraph for you: an LLM node, a tool node, and the conditional edge that loops between them. It works best with roughly 5–15 well-described tools; beyond ~20 the model starts picking the wrong one.

2.4 Day 2 Build: A Stateful LangGraph Agent

Start with the prebuilt agent plus a checkpointer, so it remembers across turns. Save as day2_graph.py:

from dotenv import load_dotenv
from langchain_openai import ChatOpenAI
from langgraph.prebuilt import create_react_agent
from langgraph.checkpoint.memory import InMemorySaver

load_dotenv()

# --- A tool is just a typed Python function ---
def get_weather(city: str) -> str:
    """Get current weather for a city."""
    return f"{city}: 21C, clear"

model = ChatOpenAI(model="gpt-4o-mini", temperature=0)
checkpointer = InMemorySaver()

agent = create_react_agent(
    model,
    tools=[get_weather],
    checkpointer=checkpointer
)

# thread_id ties messages to a persistent conversation
config = {"configurable": {"thread_id": "user-123"}}

def ask(text):
    result = agent.invoke(
        {"messages": [{"role": "user", "content": text}]},
        config
    )
    print(result["messages"][-1].content)

ask("What's the weather in Lisbon?")
ask("What about Madrid?")
ask("Which of the two was warmer?")  # needs memory of both

The third question only works because the checkpointer kept the earlier turns. Now see what the prebuilt agent hides — the same loop as an explicit graph:

from typing import Annotated, TypedDict
from langgraph.graph import StateGraph, START, END
from langgraph.graph.message import add_messages
from langgraph.prebuilt import ToolNode, tools_condition

# 1. State: a list of messages that nodes append to
class State(TypedDict):
    messages: Annotated[list, add_messages]

model_with_tools = model.bind_tools([get_weather])

# 2. Nodes
def call_model(state: State):
    return {"messages": [model_with_tools.invoke(state["messages"])]}

# 3. Wire the graph
graph = StateGraph(State)
graph.add_node("model", call_model)
graph.add_node("tools", ToolNode([get_weather]))
graph.add_edge(START, "model")
# conditional edge: tool call? -> tools, else -> END
graph.add_conditional_edges("model", tools_condition)
graph.add_edge("tools", "model")  # loop back

app = graph.compile(checkpointer=checkpointer)

What to observe

The explicit graph is the Day 1 while True loop drawn as nodes and edges. model → tools → model is the cycle; tools_condition is the "done?" check. Once you see that create_react_agent is just this graph pre-wired, frameworks stop being magic.

Day 03 Systems: MCP, A2A & Multi-Agent

3.1 The Interoperability Problem

Your agent can use tools and remember things. Now it has to live in a world with other tools and other agents. Two recurring problems show up:

  • Agents need tools and data. If every tool needs custom glue for every agent, you get an N×M integration mess.
  • Agents need to talk to other agents. A research agent built by one team needs to hand a task to a billing agent built by another.

Two open protocols emerged for these two jobs, and as of late 2025 both sit under the Linux Foundation's Agentic AI Foundation, making them vendor-neutral:

MCP — Model Context Protocol ── agent ↔ tools A2A — Agent-to-Agent Protocol ── agent ↔ agents they are COMPLEMENTARY, not competing A2A handles orchestration · MCP handles tool execution

3.2 MCP — Model Context Protocol

The pitch: "USB-C for AI." Before MCP, connecting an agent to Gmail, a database, and web search meant three bespoke integrations. MCP standardizes the plug — write a server once, any MCP-compatible host can use it. Anthropic introduced it in November 2024; by 2026 its SDKs see on the order of 97 million downloads a month.

Architecture: host → client → server

HOST (the LLM app: Claude Desktop, Cursor, your agent) │ ├── CLIENT (one connector per server, lives in the host) │ │ JSON-RPC over stdio or Streamable HTTP │ ▼ └──► SERVER (exposes capabilities to the model)

Server primitives — what a server exposes

  • Tools: executable actions the model can call (search, send, query).
  • Resources: read-only data the host can load as context (files, records).
  • Prompts: reusable, parameterized prompt templates.

Client primitives — "advanced MCP"

This is what turns a server from a passive toolbox into an active participant in the loop:

  • Sampling: the server can ask the host's model for a completion mid-execution — to reason about intermediate state or generate content. The user can review it first.
  • Elicitation: the server can pause and ask the user for input (a missing field, a confirmation) without hardcoding it.
  • Roots: the client tells the server which filesystem locations it's allowed to touch.

Transports

stdio for local servers (the host spawns the server as a subprocess). Streamable HTTP for remote servers — it replaced the older HTTP+SSE transport, which is now deprecated. The 2026 spec direction pushes a stateless core plus extensions like Tasks (long-running work) and MCP Apps (server-rendered UI).

3.3 A2A — Agent-to-Agent Protocol

MCP explicitly does not solve agent-to-agent coordination. That's A2A's job. Built by Google (April 2025), donated to the Linux Foundation (June 2025), and stabilized at v1.0 in 2026 with 150+ organizations running it in production.

How it works

  • Agent Card: each agent publishes a JSON manifest (typically at /.well-known/agent.json) describing its identity and skills. v1.0 added cryptographic signatures so you can verify who you're talking to.
  • Discovery: a client agent reads another agent's card to learn what it can do.
  • Task exchange: the client sends a task over JSON-RPC; the two exchange messages and artifacts and track the task through states (submitted → working → completed / failed).

MCP vs A2A — the comparison you'll be asked

MCPA2A
ConnectsAgent ↔ tools & dataAgent ↔ other agents
Question it answers"How does my agent use a tool?""How does my agent delegate to another agent?"
Discovery unitTools / resources / promptsAgent Card (skills)
Wire formatJSON-RPCJSON-RPC
GovernanceLinux Foundation (AAIF)Linux Foundation (AAIF)

The production answer is "both." Individual agents use MCP to reach their tools; agents use A2A to coordinate with each other. Deploying both is increasingly the expected baseline.

3.4 Multi-Agent Architectures

One agent isn't always enough. Reach for multiple when: a single agent has too many tools (past ~15–20 it picks wrong), the work spans distinct skill domains, or separate context windows keep each agent focused.

PatternShapeBest for
SupervisorOne router delegates to specialists, synthesizesClear task division (research / write / review)
SwarmPeer agents hand off control to each otherDecentralized flows where any agent may take over
HierarchicalSupervisors of supervisorsLarge systems with team-of-teams structure
PipelineFixed sequence of agentsStable, ordered stages

The live debate

Multi-agent is not free. Every extra agent adds latency, token cost, and a new failure surface, and shared context gets harder to keep coherent. There's a real industry split — some teams publish multi-agent research systems, others argue "don't build multi-agents" and keep a single strong agent with good tools. The honest answer: single agent until it visibly fails, then the smallest multi-agent topology that fixes the failure. (Deliverable #2 goes deep on this debate.)

3.5 Day 3 Build: A Supervisor Multi-Agent System (Your Demo)

A supervisor routing between two specialists. Save as day3_multiagent.py — this is your demo artifact:

from dotenv import load_dotenv
from langchain_openai import ChatOpenAI
from langgraph.prebuilt import create_react_agent
from langgraph_supervisor import create_supervisor

load_dotenv()
model = ChatOpenAI(model="gpt-4o-mini", temperature=0)

# --- Specialist 1: math ---
def add(a: float, b: float) -> float:
    """Add two numbers."""
    return a + b

def multiply(a: float, b: float) -> float:
    """Multiply two numbers."""
    return a * b

math_agent = create_react_agent(
    model, tools=[add, multiply],
    name="math_expert",
    prompt="You are a math expert. Use one tool at a time."
)

# --- Specialist 2: research (stub the tool for the demo) ---
def web_search(query: str) -> str:
    """Search the web for information."""
    return "Lisbon population is about 550,000."

research_agent = create_react_agent(
    model, tools=[web_search],
    name="research_expert",
    prompt="You are a researcher. Look things up, don't do math."
)

# --- Supervisor routes between them ---
workflow = create_supervisor(
    [research_agent, math_agent],
    model=model,
    prompt="You manage a researcher and a math expert. "
           "Delegate each subtask to the right one."
)
app = workflow.compile()

result = app.invoke({"messages": [{
    "role": "user",
    "content": "Find Lisbon's population, then multiply it by 3."
}]})

for m in result["messages"]:
    m.pretty_print()  # shows the routing — keep on for the demo

Demo tip

pretty_print() shows the supervisor handing the lookup to research_expert, then the multiplication to math_expert, then composing the answer. That hand-off trace is your narrative: "here the supervisor decides who's responsible for each subtask." That's the whole pitch of multi-agent in one screen.

Stretch: expose a tool as an MCP server

To make the protocol real, wrap a tool in an MCP server with FastMCP. Save as day3_mcp_server.py:

from mcp.server.fastmcp import FastMCP

mcp = FastMCP("weather")

# the @tool decorator = a Tool primitive other hosts can call
@mcp.tool()
def get_weather(city: str) -> str:
    """Get current weather for a city."""
    return f"{city}: 21C, clear"

if __name__ == "__main__":
    mcp.run()  # speaks MCP over stdio

Point any MCP-compatible host (Claude Desktop, Cursor, or a LangGraph MCP client) at this file and get_weather shows up as a tool — no per-host integration code. That's the entire value of the protocol in 12 lines.

3.6 What to Say in the Interview

"Most things people call agents are really workflows — fixed code paths with LLM calls inside. I reach for a true agent only when the control flow genuinely can't be written ahead of time. Starting simple is a feature, not a compromise."

On workflows vs agents

"A raw tool loop is fine until you need memory, human approval, or resume-after-crash. LangGraph gives you that by modeling the agent as a graph with checkpointed state — the loop becomes nodes and conditional edges you can inspect and interrupt."

On LangGraph

"MCP and A2A aren't competitors — they're different layers. MCP is how an agent reaches its tools; A2A is how agents delegate to each other. Production systems run both: A2A for orchestration, MCP for execution. Both are JSON-RPC, both are under the Linux Foundation now."

On protocols

"Multi-agent buys you focus and separation of concerns, but it costs latency, tokens, and a bigger failure surface. I default to a single strong agent and only split when one agent is visibly drowning — too many tools or genuinely distinct skill domains."
